Chapter 7: Software Engineering 

Software Engineering The branch of computer science that seeks principles to guide the development of large, complex software systems.
Integrated Development Environment (IDE) A system that combines tools for delveloping software (editors, compilers, debugging tools, and so on) into a single, integrated package.
Assocition for Computing Machinery (ACM)

Software Life Cycle: Development phase
  1. Requirements analysis: The goal of this first phase is to specify what services the proposed system will provide, to identify any conditions (time constraints, security, and so on) on those services, and to define how the outside world will interact with the system.
  2. Design: This phase involves creating a plan for the construction of the proposed system. Developing a solution to the problem put forth during requirement analysis.
  3. Implementation: This phase involves the actual writing of programs, creation of data files, and development of databases.
  4. Testing: In the traditional development phase of the past, testing was essentially equated with the process of debugging programs and confirming that the final software product was compatible with the software requirements specification. Today, this vision of testing is considered far too narrow. It is thought that testing should be incorporated into each phase of the Software Life Cycle.
Modular implementation: When dealing with larger complex software, the implementation can be broken up into modules (or isolated functions) in order to make the implementation process more manageable.
Unified Modeling Language (UML): A modern collection of tools developed with the object-oriented model in mind.
Use case: interactions between the system and its user within the use case diagram.
Use case diagram: This diagram depicts the proposed system as a large rectangle in which interactions (use cases) between the system and its users are represented as oval and user of the system (actors) are represented as stick figures (even though the actor may not be a person).
Data relationships:
  1. one-to-one: a relationship where there is one class and one association.
  2. One-to-many: a relationship where there is one class and many associations.
  3. many-to-many: a relationship where there is many classes and many associations.
Design pattern: a pre-developed model for solving a recurring problem in software design.
Software Quality Assurance (SQA): a group of people within a software company tasked with overseeing and enforcing the quality control systems adopted by the organization.
Software Testing:
  1. Glass box testing: a method of testing where the software tester is aware of the interior structure of the software and uses this knowledge when designing the test.
  2. Black box testing: a method of testing that does not rely on knowledge of the software’s interior structure.
  3. Beta testing: a type of black box testing in which a preliminary version of the software is given to a segment of the intended audience with the goal of learning how the software perform in real-life situations before the final version of the product is solidified and released to the market.
  4. Alpha testing: similar to beta testing except this testing is performed at the developer’s site.
Documentation:
  1. User documentation: documentation prepared for the user, intending to explain the features of the software and describe how to use them.
  2. System documentation: is to describe the software’s internal composition so that the software can be maintained later in its life cycle.
  3. Technical documentation: is to describe how a software system should be installed and serviced (such as adjusting operating parameters, installing updates, and reporting problems back to the software’s developer).
Intellectual property law:
  1. software license: a legal agreement between the owner and user of a software product that grants the user certain permissions to use the product without transferring ownership rights to the intellectual property.
  2. Software disclaimer: a legal declaration within the software license that protects software developers from certain liabilities.
Best defenses against software liability claims: is to apply sound software engineering principles during the software’s development, to use a level of care compatible with the software’s application, and to produce and maintain records that validate these endeavors.
